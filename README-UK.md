### Мотивація розробки проєкту

#### Біль
Під час вивчення дисципліни __`Бази Даних і Інформаційні системи`__ деякі мої одногрупники були вимушені після встановлення SQLite суттєвий проміжок часу налаштовувати його (особливо це стосувалося користувачів ОС Windows, через методологію встановлення ПЗ). 
SQLite - є однією з небагатьох СУБД, яка працює безпосередньо з файлом бази даних без необхідності піднімати сервер, на відміну від MySQL, PostgreSQL та інших. Та навіть налаштування такої, за ідеєю простої в налаштуванні, розгортанні та роботі, СУБД вимагає деяких навичок, часу та, найголовніше, апаратного забезпечення з повноцінною ОС (Windows, Linux, MacOS або інших) з правами адміністратора для встановлення. Користувачам мобільних пристроїв з ОС Android або IOS взагалі встановити цю СУБД напряму - неможливо. Звісно, є інтернет сервіси, що надають доступ для роботи з різними реляційними СУБД (в тому числі і з SQLite), вони вирішують проблему з кроссплаформеністю, проте створюють нові - залежність від стабільного інтернет зʼєднання (для відправки команд на сервер, який в свою чергу надсилає їх своїй СУБД) та найгірше - ізольованість даних сесії, необхідність раєстрації, фундаментальна неможливість збереження конфеденційності введених даних (через відправлення їх на сервер) - як наслідок відсутність збереження даних або чутливість даних до злому.

Коротко про біль: все ще деякий порог входу (базові навички роботи з командним рядком), час на встановлення на налаштування СУБД, необхідність ПК... ==(можливо ще якісь, згадати)==

#### Підґрунтя для вирішення проблеми (стек технологій)

Виходячи з філософії творців SQLite - як СУБД, що вимагає мінімальних налаштувань для роботи (без серверу) та мого особистого болю користування було взято за мету:
* Повністю прибрати етап встановлення та налаштування СУБД SQLite користувачем. Встановлення та підключення відбувається програмно під час завантаження сторінки.
* Зробити застосунок кроссплатформенним (у вигляді веб сайту - працює будь де, де є сучасний браузер)
* Додати можливість збереження доданих даних у БД без необхідності реєстрації та відправлення їх на сервер. (в якості файлової системи браузера - технологія OPFS)
* Додати інтеграцію з віддаленими БД на базі SQLite, наприклад Cloudflare D1. На відміну від схожих веб застосунків - це є опцією, а не обовʼязковим елементом.
* Можливість роботи повністю локально (необхідність в інтернет підключенні лише на етапі завантаження файлів веб сторінки)
* Створити простий та зрозумілий інтерфейс для користувача. Серед основних: підсвітка синтаксису SQL (завдяки бібліотеці highlight.js), автодоповнення на основі минулих відправлених команд, навігація кнопками, тумблерами, адаптивна колірна тема.
* Безпека. Режими Safe та Read only можуть бути активовані користувачем для унеможливлення випадкового відправлення потенційно небезпечних команд.
> [!warning]
> *OPFS може не підтримуватися на старих версіях веб браузерів, що унеможливлює локальне збереження внесених змін в БД всередині ізольваної файлової системи OPFS, проте це не впливає на роботу з БД, поки дані є в оперативній пам'яті під час сесії*

==Знайомство з реляціними БД (SQLite як основної) та мій досвід роботи з екосистемою Cloudflare (D1 як основа для API віддаленої реляційної БД).==

### Чому BravSQL
### Цільова аудиторія 
- **Школярі та студенти** — навчання SQL без встановлення ПЗ, особливо ті, хто не має власного ПК
- **Розробники** — швидке тестування SQL запитів на будь-якому пристрої, можливість підключення реальної бази даних за допомогою [Cloudflare D1 сумісного API](#cloudflare-d1-сумісний-api-для-працювання-з-базами-даних), використання застосунку як захисного шару з приємним інтерфейсом.
- **Звичайні користувачі** — ознайомлення з СУБД SQLite без технічних знань
- **DevOps інженери** — безпечне тестування міграцій перед production завдяки можливості копіювання частини записів з віддаленої таблиці або копіювання її повністю у локальну базу даних
- **Викладачі** — демонстрація SQL без налаштування середовища для кожного студента, виконання завдань прямо на сайті, відсутність необхідності прав адміністратора на пристрої.
### Архітектура застосунку

#### Стек технологій
HTML, CSS, JavaScript/TypeScript 
Web Workers, WASM, OPFS
#### Головні опції
**Режими роботи:**
Локальний/глобальний, захищений/незахищений, read only/no read only.
**Налаштування:**
підключення віддаленої СУБД з Cloudflare D1 сумісним API, клонування частини/всієї таблиці з віддаленої бд до локальної (наприклад для безпечних експериментів), веб адреса проксі сервера - використовується для обходу CORS обмежень ( є проксі за замовчуванням ), вибір шляху до локальної бази даних sqlite (будь яка назва, окрім назви, що відноситься до даних клонованих з віддаленої БД), ==збереження дійсних шляхів до локальних і глобальних баз даних в localStorage і їх відображення десь у вигляді переліку з активними посиланнями з можливістю натискання на якийсь з шляхів для перемикання на нього==

```json
}
```
